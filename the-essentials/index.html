The Essentials

Before we get started on this documentation journey, we need a basic understanding of what documentation is, and why we care about it.

What is Documentation?

Documentation is what gives the software you write a voice. Code can be leveraged to do incredible things. However, even in the most readable of programming languages, code cannot easily:

  • Tell a potential user what they can gain by using your software
  • Fully brief a contributor on how to set up the correct development environment and deploy the backend properly
  • Convince a hackathon judge that your project is innovative and can have a real-world impact

Documentation speaks where code can't, so it's important to have at least some resources available for the different types of people that will interact with your project.

Why Should We Document?

1. You (want to be) using your code in 6 months

Code written 6 months ago is almost indistinguishable from code written by someone else. If you want to continue iterating on your project after the hackathon, you'll want to be spending time adding new flashy features and making things scalable, not untangling a mess of code that you don't understand or wasting time figuring out how to set up a development environment again.

2. You want people to use your code

As mentioned earlier, your project's code doesn't do a great job of explaining why it exists. You've written a piece of code and released it into the world because you think it will be useful to others. However:

  • If people don't know why your project exists, they won't use it
  • If people can't figure out how to install your code, they won't use it
  • If people can't figure out how to use your code, they won't use it

Documentation is how we ensure our code can be used and appreciated by the world.

Note: These points were very heavily inspired by this excellent presentation by Eric Scholscher in 2013.

What Will Our Documentation Look Like?

So far, we've been referring to "documentation" as a single entity or thing. However (as you'll soon see on the next page), documentation encompasses a broad variety of things in a variety of forms, that all share the common goal of explaining what your code does, how it's used, or how it can be leveraged.

For the purposes of this guide, there are two places we'll be creating documents: in our project's Git repository and on a dedicated docs site (with it's own repository).
